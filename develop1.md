# 文档一：整体设计思路

## 1. 项目定位
本工具为工业网关的上位机调试与配置软件，主要用于管理网关内部的协议采集、数据处理、转发规则等配置。支持多用户（工程师）同时编辑同一份配置，提供完整的配置版本管理、实时同步、冲突检测等功能。

## 2. 架构选型原则
- 本地部署，前后端分离：后端服务和前端界面均运行在同一台 PC 上，通过本地 HTTP 接口通信。无需外部服务器，便于现场部署和离线使用。
- 数据集中存储：采用 SQLite 作为本地数据库，避免文件共享带来的并发冲突，通过后端统一管理数据访问。
- 多用户协作基础：虽为本地部署，但设计上预留了后期迁移到远程服务器的能力（同一套 API 接口），支持多个上位机客户端同时连接同一个 SQLite 数据库文件（通过后端服务管理并发）。
- 热更新能力：复杂图表和实时监控界面采用 QML + WebView 混合技术，支持从本地或远程服务器动态加载 UI 资源，满足内部迭代频繁调整布局的需求。

## 4. 关键设计决策
- 前后端分离的目的：即使在同一台机器上，分离也能够让前端只关心界面和交互，后端负责数据持久化、并发控制、实时推送。后期若需迁移到服务器，只需修改前端 API 地址即可。
- SQLite 的选择：单文件、零配置、支持并发读/单写（后端服务作为唯一写者），完全满足本地多用户同时编辑的需求。配合 WAL 模式和适当的繁忙超时，可支持多客户端同时连接。
- 实时同步机制：后端通过 WebSocket 将配置变更推送给所有在线前端，保证协作编辑时界面实时刷新。
- 冲突处理：采用乐观锁（每个记录带 version 字段），更新时检查版本，提供 409 冲突响应。
- 热更新：复杂图表区域使用 QML + WebView 动态加载，支持系统管理员通过后端 API 上传新版 QML 资源包，客户端根据版本号自动下载并替换，无需升级整个应用程序。

## 5. 数据流简述
1. 前端启动 → 连接本地后端（默认 localhost:8000）→ 拉取全量配置（若本地有缓存则先显示缓存）→ 建立 WebSocket 长连接。
2. 用户编辑某个采集点 → 前端立即更新界面（乐观更新），同时发送 PUT 请求到后端。
3. 后端接收请求，检查版本号，更新数据库，成功后通过 WebSocket 广播变更。
4. 其他在线前端收到广播 → 重新拉取该记录并刷新界面。
5. 若更新时版本冲突 → 后端返回 409，前端自动拉取最新数据并提示用户。

## 6. 部署与运行方式
- 后端服务以独立进程形式启动（可做成 Windows 服务或命令行运行）。
- Qt 前端为独立可执行文件，启动时自动检测后端是否运行（若无则提示启动或自动拉起）。
- 所有组件均安装在同一 PC 上，无需网络连接即可工作（除 QML 热更新需要可选的服务器外）。

## 7. 未来演进路径
- 第一阶段（当前）：本地 SQLite + 本地后端 + Qt 前端，单机协作。
- 第二阶段：将后端部署到局域网服务器，多台 PC 上的 Qt 前端通过网络连接同一后端，实现远程协作。
- 第三阶段：后端集群化、数据库升级为 PostgreSQL，支持更大规模的并发和分布式部署。
