# 文档三：后端设计思路

## 1. 技术选型
- 语言：Python 3.10+
- Web 框架：FastAPI（异步、自动文档、性能优秀）
- 数据库驱动：`sqlite3`（Python 标准库）+ `aiosqlite`（异步支持）
- WebSocket：FastAPI 内置支持
- 数据校验：Pydantic
- 并发控制：`asyncio` + 数据库连接池，乐观锁由业务代码实现
- 文件服务：提供 QML 资源包的版本管理和下载

## 2. 服务架构
后端作为独立进程运行，提供以下能力：

- REST API 服务（核心配置 CRUD）
- WebSocket 服务（实时推送配置变更）
- 静态文件服务（为前端提供 QML 资源包下载）
- 可选：集成 SQLite 的访问管理（单写多读）

## 3. API 设计原则
- RESTful 风格，资源路径清晰，如：
  - `/api/v1/points`
  - `/api/v1/modules`
  - `/api/v1/rules`
- 所有写操作（`POST`/`PUT`/`DELETE`）要求请求体携带 `version` 字段。
- 批量操作：支持 `PATCH /api/v1/points/batch`，部分成功时返回冲突列表。
- 统一错误响应格式，使用 HTTP 状态码：
  - `200` 成功
  - `400` 请求参数错误
  - `404` 资源不存在
  - `409` 版本冲突
  - `500` 服务器内部错误

## 4. 数据并发控制详细设计
### 4.1 乐观锁流程
1. 客户端 `GET` 资源时，响应中包含 `version` 字段。
2. 客户端 `PUT` 请求时，将 `version` 原样带回。
3. 后端执行 `UPDATE ... WHERE version = old_version`。
4. 若更新行数为 `0`，则查询当前最新版本，返回 `409` 响应并携带最新版本信息。
5. 客户端收到 `409` 后，重新 `GET` 最新数据，提示用户并可选择自动合并。

### 4.2 批量更新冲突处理
- 对批量更新逐条执行。
- 遇到冲突时不回滚整个批次。
- 返回成功项与失败项（冲突项）列表。

## 5. 实时推送（WebSocket）设计
### 5.1 连接管理
- 维护全局 `active_connections` 列表。
- 每个客户端连接成功后加入列表。

### 5.2 推送时机
- 任意成功的数据变更操作（增、删、改）之后，立即向所有连接广播变更消息。
- 广播内容包含：表名、操作类型、记录 ID、新版本号。

### 5.3 消息格式
使用轻量 JSON，例如：

```json
{"table":"points","op":"update","id":123,"new_version":5}
```

### 5.4 客户端处理策略
- 客户端收到广播后，若该资源当前在 UI 中可见，则主动调用 `GET` 接口刷新最新数据。

## 6. QML 热更新支持
### 6.1 资源包管理
- 后端维护 `qml_versions` 表，记录版本号、文件路径、上传时间。

### 6.2 API
- `GET /api/qml/version`：返回最新版本号。
- `GET /api/qml/download?version=xxx`：返回指定版本的 ZIP 包。

### 6.3 更新流程
1. 管理员通过独立工具上传新 QML 资源包到后端指定目录，并更新版本表。
2. 客户端启动时或定期检查版本号。
3. 发现新版本后下载并解压，刷新 QML 界面。

## 7. 启动与配置
- 后端默认监听 `127.0.0.1:8000`，可通过配置文件修改端口。
- 数据库文件路径可配置（默认 `./gateway_config.db`）。
- 日志输出到文件和控制台，便于调试。

## 8. 安全性（内部使用）
- 同机运行场景下，可默认不启用认证（信任本地访问）。
- 若需远程访问，后续可增加 JWT 认证。
- CORS 仅允许 `localhost` 或指定 IP。

## 9. 错误处理与日志
- 使用 `loguru` 记录关键操作（如增删改、WebSocket 连接、冲突事件）。
- API 异常统一由全局异常处理函数捕获，返回友好的错误信息。

## 10. 测试策略
- 单元测试：模拟数据库操作，测试乐观锁逻辑。
- 集成测试：启动测试后端，使用 `httpx` 调用 API，验证并发冲突场景。
