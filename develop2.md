# 文档二：数据库设计思路

## 1. 数据库选型
采用 SQLite 3 作为持久化存储引擎，原因如下：

- 轻量级，无需单独安装和管理，随服务一起发布。
- 支持 ACID 事务，满足配置数据一致性要求。
- 支持并发读和单写，后端作为唯一写入进程，通过连接池管理多前端并发读请求。
- 单文件便于备份、恢复和版本控制。

## 2. 核心实体关系
数据库需要表达以下业务概念：

- 设备实例：表示一个物理或逻辑设备（如 DSP1、IO 模块、本地 ARM 设备）。
- 数据点：全局唯一信号定义（`sig_id`），包含数据类型、大小、处理函数等信息。
- 采集配置：针对 Modbus 或 UPP 协议的采集规则，关联到特定数据点。
- 转发规则：将内部数据点映射到外部 Modbus 寄存器，供上位机查询。
- UPP 模块：UPP 协议中以模块为单位组织多个采集点。

## 3. 主要表结构概览
### 3.1 设备表（`devices`）
存储每个设备的标识、通信端口、采集地址等。每个设备有唯一 `device_code`（对应 C 代码中的枚举值）。

### 3.2 数据点表（`data_points`）
核心字典表，记录所有信号 ID（`sig_id`）、数据类型、长度、回调函数名等元数据。此表与采集配置、转发规则、本地处理点均有关联。

### 3.3 采集配置表（`acquisition_configs`）
支持多种协议：

- Modbus 协议：包含寄存器类型、地址、数量。
- UPP 协议：包含模块 ID、数据偏移、长度、读写模式等。

通过 `protocol` 字段区分，不同协议字段使用独立列（允许 `NULL`），便于扩展。

### 3.4 设备采集关联表（`device_acquisition`）
将采集配置分配到具体设备实例。每个设备可有多个采集点。

### 3.5 转发规则表（`forwarding_rules`）
定义从内部数据点到外部 Modbus 寄存器的映射，包含目标信号 ID、寄存器类型、起始地址、寄存器数量及优先级。

### 3.6 设备转发关联表（`device_forwarding`）
将转发规则分配给设备（常用于虚拟设备，如 ARM 对外提供的 Modbus 从站）。

### 3.7 设备本地点表（`device_local_points`）
记录不通过采集，而由设备自身维护的信号（如串口配置、AES 密钥等），关联设备与数据点。

### 3.8 UPP 模块表（`upp_modules`，可选）
用于管理 UPP 模块元数据，便于前端按模块树形展示采集点。

## 4. 并发控制设计
- 乐观锁：所有可编辑业务表（设备、数据点、采集配置、转发规则）均包含 `version` 整数字段，初始值为 `1`。更新时执行 `UPDATE ... WHERE id = ? AND version = ?`，根据受影响行数判断是否冲突。
- 版本维护：由后端自动递增，前端在修改前先获取当前版本号。

## 5. 变更通知机制
可采用以下任一方式：

- 使用 SQLite 触发器（或后端业务逻辑）在内容变更时记录到 `change_log` 表。
- 直接通过 WebSocket 推送变更。

由于 SQLite 不支持原生 `NOTIFY`，建议由后端应用层在每次成功写入后推送变更消息。

## 6. 数据完整性约束
- 外键约束：确保 `device_acquisition` 引用的设备和采集配置存在。
- 唯一约束：
  - `data_points.sig_id` 唯一。
  - `acquisition_configs` 中针对 Modbus 的 `(reg_type, reg_addr)` 组合在全设备范围内应唯一（由后端保证）。
- 级联删除：谨慎使用，通常采用 `ON DELETE RESTRICT`，防止误删被引用记录。

## 7. 性能优化要点
- 为主键、外键列以及 `version` 列建立索引。
- 多表关联查询（如按设备查询所有采集点）可使用视图或预先 `JOIN` 降低查询复杂度。
- 启用 SQLite 的 WAL 模式，提高并发读写性能。

## 8. 扩展性考虑
- 预留 `attributes` JSON 字段，用于存储协议相关扩展属性（如未来新增 CANopen 协议参数）。
- 表结构支持多网关管理：可增加 `gateway_id` 列，实现一个数据库管理多个网关配置。
